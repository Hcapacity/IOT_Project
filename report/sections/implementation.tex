\section{Implementation}

\subsection{System Startup and Resource Allocation}
The entry point of the application is intentionally lightweight. In
\texttt{main.cpp}, the \texttt{setup()} function performs four steps:
\begin{enumerate}
  \item initialize the serial port and I\textsuperscript{2}C bus,
  \item allocate the application context,
  \item create queues, mutexes, and the Internet-ready semaphore, and
  \item spawn all RTOS tasks.
\end{enumerate}

This startup strategy is important because it keeps initialization
deterministic. The \texttt{loop()} function does not perform application logic;
it simply yields with a long delay. In other words, after boot the entire
system behavior is task-driven. This is a clear and appropriate RTOS pattern
for the given assignment.

The application context is initialized by \texttt{init\_app\_shared\_state()},
which sets default values such as invalid sensor readings, disconnected Wi-Fi
state, and the default LCD content mode. This avoids undefined startup
behavior.

\subsection{Shared State Helpers}
The file \texttt{global.cpp} implements helper functions to read and update the
compact shared state block. The core logic is simple but useful:
\begin{itemize}
  \item \texttt{lockSharedState()} and \texttt{unlockSharedState()} hide mutex
  handling details,
  \item \texttt{app\_set\_latest\_sensor()} updates the latest temperature and
  humidity,
  \item \texttt{app\_set\_wifi\_connected()} and
  \texttt{app\_get\_wifi\_connected()} manage connectivity state, and
  \item \texttt{app\_set\_lcd\_content\_mode()} and
  \texttt{app\_get\_lcd\_content\_mode()} coordinate LCD display policy.
\end{itemize}

The value of this module is not its complexity but its role in improving code
quality. It makes synchronization explicit and reduces the need for broad
\texttt{extern} globals. This is a meaningful improvement over directly
exposing mutable state to every file.

\subsection{Sensor Task}
The sensor task is the timing anchor of the whole application. It performs
periodic DHT20 acquisition every 2 seconds using \texttt{vTaskDelayUntil()}.
This API is a good design choice because it maintains a stable period relative
to the previous wake time, instead of accumulating drift through repeated
\texttt{delay()} calls.

The execution flow is as follows:
\begin{enumerate}
  \item initialize the DHT20 once, under I\textsuperscript{2}C mutex
  protection;
  \item on each period, attempt to lock the I\textsuperscript{2}C bus for up to
  300 ms;
  \item read temperature and humidity;
  \item if the data are valid, update the shared state and dispatch the sample
  to all consumers;
  \item if the data are invalid, propagate an invalid sample to the LCD and web
  server for graceful failure handling.
\end{enumerate}

One of the most elegant aspects of this task is the fan-out pattern. A single
physical read is turned into multiple queue writes:
\texttt{lcdQueue}, \texttt{webQueue}, \texttt{coreQueue}, and
\texttt{tinyMLQueue}. This means the sensor is sampled exactly once per period,
while the rest of the system remains modular.

\subsection{LED Manager Task}
The discrete LED task is implemented as an event-driven state machine. It does
not read sensors directly. Instead, it receives \texttt{led\_command\_t}
messages through \texttt{ledQueue}. The LED task supports two categories of
behavior:
\begin{itemize}
  \item continuous blinking according to the current temperature mode, and
  \item short pulse sequences for exceptional events.
\end{itemize}

The temperature-to-blink mapping is implemented through
\texttt{classify\_temperature\_mode()} and \texttt{led\_half\_period\_ticks()}:
\begin{itemize}
  \item cold: 5-second half-period,
  \item normal: 1-second half-period,
  \item hot: 100-ms half-period,
  \item error: 120-ms half-period.
\end{itemize}

The task also supports quick pulse bursts:
\begin{itemize}
  \item Wi-Fi connected: 3 fast blinks,
  \item error on: 6 fast blinks,
  \item error clear: stop pulsing and return to off.
\end{itemize}

The strong point of this design is that the LED module is both periodic and
reactive. It uses queue timeouts as implicit timing intervals, so it can either
wait for a new command or toggle according to the current mode without another
timer task.

\subsection{NeoPixel Task}
The NeoPixel task is simpler but still well designed. It blocks on
\texttt{neoQueue} and updates the RGB LED whenever a new humidity command
arrives. The function \texttt{humidity\_to\_color()} maps humidity into a
continuous gradient:
\begin{itemize}
  \item \(0\%\) to \(50\%\): red to green,
  \item \(51\%\) to \(100\%\): green to blue.
\end{itemize}

This is an improvement over a strict three-bin implementation because it
provides richer visual information while still satisfying the assignment's
three-level requirement. The task also supports a disable command that clears
the LED immediately, which is useful for future extension or power-saving
scenarios.

\subsection{LCD Task}
The LCD task owns the local user interface. It maintains the most recent sensor
sample and the most recent TinyML result, then decides what to show based on
system state. The logic has three levels:
\begin{enumerate}
  \item in AP mode, always show the sensor screen;
  \item in STA mode with normal LCD content mode, show the sensor screen;
  \item in STA mode with forecast LCD content mode, show the TinyML forecast.
\end{enumerate}

This conditional behavior is an excellent example of policy separation. The
task itself decides \textit{how} to render each screen, while the rest of the
system decides \textit{which content mode} is currently active.

The LCD module also reacts to environmental context:
\begin{itemize}
  \item if the sensor data are invalid, it prints \texttt{Sensor Error};
  \item if Wi-Fi is connected, it can display signal quality in percent and a
  qualitative label such as \texttt{GOOD} or \texttt{WEAK};
  \item if TinyML results exist, it shows forecast label and rain probability.
\end{itemize}

An important implementation detail is that LCD rendering is not performed at a
fixed blind period. Instead, the task updates when a new sensor sample arrives,
when TinyML changes, when RSSI changes meaningfully, or when AP/STA mode
changes. This avoids unnecessary I\textsuperscript{2}C traffic and makes the
display more event-aware.

\subsection{Web Server Task}
The web module is the largest subsystem in the repository. It combines
\texttt{WebServer}, \texttt{Preferences}, local runtime state, and Wi-Fi mode
management. Its implementation can be divided into five functional groups.

\subsubsection*{1) AP/STA workflow.}
The board starts either from the most recently saved Wi-Fi profile or directly
in AP mode if no profile exists. The helper functions
\texttt{startApMode()} and \texttt{startStaConnect()} reset Wi-Fi state,
configure the correct mode, and update internal status flags. If STA connection
fails after 10 seconds, the system automatically returns to AP mode. The BOOT
button also forces AP mode manually.

\subsubsection*{2) Persistent Wi-Fi profiles.}
The project uses \texttt{Preferences} to store up to five Wi-Fi profiles. The
most recently used profile is kept at index 0. The code supports:
\begin{itemize}
  \item reading saved profiles,
  \item updating or inserting a profile,
  \item removing all saved profiles.
\end{itemize}
This is a practical improvement because it lets the board remember STA
credentials across restarts without hard-coding a single network.

\subsubsection*{3) AP dashboard and settings pages.}
The root page in AP mode displays:
\begin{itemize}
  \item temperature and humidity,
  \item TinyML label and probability,
  \item AP IP and Wi-Fi status,
  \item target STA SSID and STA IP,
  \item external LED control,
  \item chart history, and
  \item quick Wi-Fi scan results.
\end{itemize}
The settings page expands the Wi-Fi management workflow by providing scan,
saved-profile, and connect forms. The STA page is intentionally simpler and
acts mainly as a status page rather than a full dashboard.

\subsubsection*{4) REST endpoints.}
The task registers multiple routes:
\begin{itemize}
  \item \texttt{/} for the main page,
  \item \texttt{/settings} for Wi-Fi configuration,
  \item \texttt{/api/state} for live JSON state,
  \item \texttt{/api/scan} for Wi-Fi scan results,
  \item \texttt{/api/saved} for stored Wi-Fi profiles,
  \item \texttt{/connect} for STA connection request,
  \item \texttt{/forget} for profile reset,
  \item \texttt{/api/light/toggle} for on/off control,
  \item \texttt{/api/light/brightness} for PWM brightness control.
\end{itemize}

\subsubsection*{5) STA Mode (Cloud Integration and Remote Configuration)}
When the device successfully establishes a connection to a Wi-Fi network, the Web Server transitions to a specialized configuration portal. This interface is crucial for defining where and how the system's telemetry data is routed:
\begin{itemize}
    \item \textbf{Dynamic Broker Configuration:} The STA page provides a dedicated form for customizing the communication target. Users can input the \textbf{MQTT Broker Host (Server Name)}, \textbf{Username}, and \textbf{Password} directly through the browser. This allows the system to be pointed to the official CoreIOT platform, a private MQTT broker, or a custom **Local Server** without any code changes.
    \item \textbf{Persistence and REST API:} Configuration changes are submitted via the \texttt{/api/coreiot/config} POST endpoint. The task processes these requests and uses the \texttt{Preferences} library to save the credentials into the \texttt{WIFI\_STORE\_NAMESPACE}, ensuring the settings persist even after a power cycle.
    \item \textbf{CoreIOT Task Synchronization:} The \texttt{coreiot\_task} is designed to be reactive to these changes. When a new server host or set of credentials is saved, the web task updates the shared application context. The CoreIOT task, which is constantly monitoring the connection state, will automatically trigger a reconnection sequence to the newly specified server.
    \item \textbf{Telemetry Routing Control:} The interface also includes a toggle to enable or disable publishing. This gives the user full control over data privacy, allowing them to stop cloud transmission or redirect the stream to a local endpoint for private data logging.
\end{itemize}

\subsubsection*{6) Client-side visualization.}
The AP dashboard includes JavaScript that polls \texttt{/api/state} every 2
seconds. It updates visible values and renders a chart on a
\texttt{canvas}. A recent improvement in the code is the use of dual Y-axes for
temperature and humidity, which avoids the misleading effect of drawing both
series on a single implicit scale. The chart also displays axis labels and
relative sample times.

Two implementation highlights stand out in this module. First, the task uses a
20 ms loop period, which is short enough for responsive HTTP handling without
busy-waiting. Second, the cloud subsystem is not started blindly; instead, the
web task signals \texttt{internetSemaphore} once STA connectivity is confirmed.


\subsection{TinyML Task: Edge-Based Rain Forecasting Pipeline}

The TinyML task implements an on-device rain prediction system, shifting the analytical workload from the cloud directly to the edge. This approach minimizes latency, reduces network dependency, and demonstrates the viability of executing neural networks on resource-constrained microcontrollers. The development and deployment of this predictive model follow a comprehensive Machine Learning pipeline, spanning from offline dataset preparation to real-time RTOS integration.

\subsection*{EDA}
The predictive model is built upon a historical meteorological dataset collected in Ho Chi Minh City, encompassing continuous weather observations from 2009 to 2020. The primary objective is to perform a binary classification, categorizing the upcoming weather state as either "RAIN" (Class 1) or "SUNNY/SAFE" (Class 0).

\textbf{Dataset Overview:} The dataset comprises 34,976 valid samples with zero missing values across all features. The temporal span extends from January 1, 2009, to December 20, 2020, representing a complete 4,372-day period with 100\% data coverage.

\subsection*{Feature Engineering and Extraction}
The raw dataset contains the original weather observation fields, but the model is trained on a compact numerical representation derived from \texttt{Temp}, \texttt{Humidity}, \texttt{Date}, and \texttt{Time}. The preprocessing pipeline first extracts numerical temperature and humidity values, then converts calendar information into cyclical time features, and finally constructs a feature vector that captures both the current state and short-term change patterns.

The exact feature engineering steps are implemented as follows:
\begin{enumerate}
  \item \textbf{Temperature and humidity extraction:} \texttt{Temp\_c} is extracted from the \texttt{Temp} string field, while \texttt{Hum\_p} is extracted from \texttt{Humidity}. This converts the raw text measurements into numeric values usable by the neural network.
  \item \textbf{Temporal encoding:} The month is mapped from the date string and the hour is parsed from the time string. Instead of using raw month and hour values directly, the model encodes them with sine and cosine functions to preserve their circular nature.
  \item \textbf{Short-term deltas:} \texttt{Temp\_diff} and \texttt{Hum\_diff} are computed using the first-order difference between consecutive samples. These variables represent local trend changes, which are useful for capturing sudden weather transitions.
  \item \textbf{Final input vector:} The model uses eight engineered features: \texttt{Temp\_c}, \texttt{Hum\_p}, \texttt{Temp\_diff}, \texttt{Hum\_diff}, \texttt{Month\_sin}, \texttt{Month\_cos}, \texttt{Hour\_sin}, and \texttt{Hour\_cos}.
\end{enumerate}

This design is appropriate for the dataset because it combines instantaneous weather state with trend information and periodic time context. The scaling step using \texttt{StandardScaler} is then applied only on the training split, ensuring that all features contribute to learning on a comparable numerical scale.

\noindent\textbf{Evaluation of the EDA figures.} The descriptive statistics figure shows that the engineered variables are well formed and fully populated across 34,976 samples. The central values are physically plausible: temperature is concentrated around the high-20s Celsius, humidity is centered in the mid-70\%, and the temporal features have means close to zero after sine-cosine encoding, which is expected for cyclical variables. The spread of \texttt{Temp\_diff} and \texttt{Hum\_diff} is also moderate, indicating that the dataset contains both stable periods and noticeable transitions rather than being dominated by flat readings.

The class-distribution chart shows a clear imbalance, with Class 0 accounting for 78.8\% and Class 1 accounting for 21.2\%. This means the problem is not perfectly balanced, so accuracy alone is not sufficient to judge model quality. The imbalance also explains why the model should be evaluated with additional measures such as precision, recall, and F1-score, especially for the rain class.

The correlation heatmap confirms that no single feature has an overwhelmingly strong linear relationship with the target label. The label correlates only weakly with the engineered inputs, which is a useful result because it suggests the rain pattern is not trivially separable by one variable alone. At the same time, the heatmap shows meaningful dependencies among features, such as the strong negative relationship between \texttt{Temp\_diff} and \texttt{Hum\_diff}, as well as the seasonal and daily structure captured by the cyclic time encodings. This supports the choice of a small neural network, since a nonlinear model is better suited than a simple linear classifier for combining weak but complementary signals.

\subsection*{Model Architecture and Training}
\noindent\textbf{Model Benchmarking and ANN Selection.} Table \ref{tab:model_benchmark} summarizes the candidate models evaluated on the same feature set. The results show that XGBoost achieved the highest accuracy, but the ANN was selected for the final ESP32 implementation because it provides competitive performance while remaining a compact deep learning model that can be quantized and deployed through TensorFlow Lite Micro.


The core logic is driven by a sequential Artificial Neural Network (ANN) designed via TensorFlow/Keras. Although the benchmark shows that XGBoost has a slightly higher accuracy, the ANN was intentionally chosen because the goal of this project is to demonstrate a deployable deep learning pipeline on the ESP32-S3. The ANN is lightweight enough for full integer quantization, yet expressive enough to capture non-linear relationships in the engineered features. The network topology consists of:
\begin{itemize}
    \item An \textbf{Input Layer} accepting the 8 standardized features.
    \item A \textbf{Hidden Dense Layer} with 12 neurons utilizing the ReLU (Rectified Linear Unit) activation function.
    \item A \textbf{Second Hidden Dense Layer} with 8 neurons, also using ReLU activation.
    \item An \textbf{Output Layer} with a single neuron applying a Sigmoid activation function, producing a final probability score between 0.0 and 1.0.
\end{itemize}

During the offline training phase, the dataset was split into an 80\% training set and a 20\% testing set. The model was compiled using the Adam optimizer and a binary cross-entropy loss function. After training for 50 epochs with a batch size of 32, the baseline model achieved an accuracy of \textbf{84.86\%} on the 32-bit floating-point (float32) test set.

\subsubsection*{4) Full Integer Quantization for Edge Deployment}
To deploy this trained model onto the ESP32-S3, it required an aggressive optimization process using the TensorFlow Lite Converter. Deploying a float32 model would consume excessive SRAM and execution time. Therefore, the model underwent \textbf{Full Integer Quantization (INT8)}.

A Representative Dataset generator (yielding 100 random training samples) was used to calibrate the dynamic range of the model's weights and activations. This calibration allowed the converter to safely map the 32-bit floats into 8-bit integers. The resulting optimized model (\texttt{.tflite}) was then converted into a C-byte array header file (\texttt{model\_data.h}) using the \texttt{xxd} utility. 

Remarkably, after this strict quantization process, the model retained an accuracy of \textbf{84.43\%}. This corresponds to an absolute drop of only \textbf{0.43 percentage points}, or about \textbf{0.51\%} relative to the original float32 accuracy. This is a highly favorable trade-off because the INT8 model drastically reduces the memory footprint and accelerates inference time on the microcontroller without utilizing floating-point hardware during the forward pass.

\subsubsection*{5) C++ Implementation in the RTOS Environment}
Within the firmware, the \texttt{tiny\_ml\_task} manages the real-time execution of this pipeline. It utilizes the \texttt{tflite::MicroInterpreter} and an \texttt{AllOpsResolver} initialized with a 16 KB tensor arena. The implementation strictly mirrors the offline training pipeline:
\begin{itemize}
    \item \textbf{Accumulation:} The task receives raw data from \texttt{tinyMLQueue} and accumulates 10 consecutive sensor samples (equivalent to a 20-second monitoring window) before triggering an inference cycle.
    \item \textbf{Normalization:} The computed 8 features are standardized locally using pre-computed \texttt{means} and \texttt{scales} arrays extracted from the Python \texttt{StandardScaler}.
    \item \textbf{Input Quantization:} The normalized float values are converted into the INT8 format required by the model using the input tensor's specific \texttt{scale} and \texttt{zero\_point} parameters.
    \item \textbf{Inference and Output:} The \texttt{interpreter->Invoke()} function is called. The resulting INT8 output is de-quantized back into a float probability. If this probability exceeds the 0.5 threshold, the system categorizes the upcoming state as "RAIN".
\end{itemize}

Finally, this prediction, along with the calculated probability percentage and timestamps, is packaged into a \texttt{tinyml\_result\_t} structure and pushed to the \texttt{tinyResultQueue}. It is subsequently consumed by the LCD task for immediate local display and by the CoreIOT task for remote cloud telemetry.

\noindent\textbf{Mathematical Formulation of the Selected ANN.} Let the final engineered input vector be
\[
\mathbf{x} = [x_1, x_2, \ldots, x_8]^T
= [\texttt{Temp\_c}, \texttt{Hum\_p}, \texttt{Temp\_diff}, \texttt{Hum\_diff}, \texttt{Month\_sin}, \texttt{Month\_cos}, \texttt{Hour\_sin}, \texttt{Hour\_cos}]^T.
\]
The selected ANN transforms this input into a rain probability through two hidden layers and one sigmoid output layer:
\[
\mathbf{h}_1 = \operatorname{ReLU}(\mathbf{W}_1\mathbf{x} + \mathbf{b}_1),
\]
\[
\mathbf{h}_2 = \operatorname{ReLU}(\mathbf{W}_2\mathbf{h}_1 + \mathbf{b}_2),
\]
\[
\hat{y} = \sigma(\mathbf{W}_3\mathbf{h}_2 + b_3),
\]
where \(\operatorname{ReLU}(z) = \max(0, z)\) and \(\sigma(z) = \frac{1}{1 + e^{-z}}\). The output \(\hat{y} \in [0,1]\) represents the predicted probability of rain. The final class decision is computed as
\[
\hat{c} =
\begin{cases}
1, & \text{if } \hat{y} \geq 0.5, \\
0, & \text{otherwise}.
\end{cases}
\]
This formulation matches the deployed ANN structure and explains how the firmware converts the 8-feature input vector into a binary weather forecast.


\subsection{CoreIOT Task: Cloud Telemetry and Remote Configuration}

The CoreIOT task is the cloud-integration layer of the entire embedded system. Its primary responsibility is to establish persistent connectivity with an external MQTT broker (either the CoreIOT cloud platform or a custom local MQTT server) and to bidirectionally stream telemetry data and remote control commands. The task is designed to be highly configurable, extensible, and reactive to network state changes, allowing the system to adapt to different deployment scenarios without firmware recompilation.

\subsection*{Initialization and Network Readiness}

The CoreIOT task does not start immediately after boot. Instead, it follows a staged initialization pattern:
\begin{enumerate}
  \item \textbf{Wait for Internet Readiness:} The task blocks on \texttt{internetSemaphore}, which is signaled by the Web Server task only after the device has successfully established Wi-Fi connectivity in STA mode.
  \item \textbf{Retrieve Configuration:} Once the semaphore is released, the task retrieves the dynamic MQTT connection parameters from persistent storage (managed by \texttt{Preferences}). This includes the \textbf{broker host}, \textbf{broker port}, \textbf{username}, and \textbf{password}.
  \item \textbf{Connect to Broker:} The task attempts to establish a TCP connection to the specified MQTT broker using the provided credentials.
  \item \textbf{Subscribe to Control Topics:} After successful connection, the task subscribes to RPC (Remote Procedure Call) request topics, enabling the cloud dashboard to send commands back to the device.
  \item \textbf{Periodic Telemetry Publishing:} Once fully initialized, the task enters a loop that publishes telemetry data every 10 seconds.
\end{enumerate}

This staged approach ensures that the cloud task does not consume resources or attempt connection during AP mode, and it allows the Wi-Fi subsystem to stabilize before the cloud subsystem is activated.

\subsection*{Dynamic Broker Configuration and Extensibility}

A key architectural feature of the CoreIOT task is its \textbf{dynamic server integration}. The MQTT broker host, port, username, and password are \textbf{not hard-coded} into the firmware. Instead, these settings are:
\begin{itemize}
  \item \textbf{Configured via the Web Interface:} When the device connects to a Wi-Fi network (STA mode), the user can navigate to the web dashboard and access the cloud settings form. This form allows users to specify:
    \begin{itemize}
      \item \textbf{MQTT Broker Host:} The IP address or fully qualified domain name (FQDN) of the target MQTT server. Examples:
        \begin{itemize}
          \item For CoreIOT Cloud: \texttt{app.coreiot.io} (default if not overridden).
          \item For a local MQTT broker running on a desktop computer in the same network: \texttt{192.168.1.100}.
        \end{itemize}
      \item \textbf{MQTT Broker Port:} Typically \texttt{1883} for standard MQTT (or \texttt{8883} for MQTT over TLS).
      \item \textbf{Username:} For CoreIOT cloud, this is the \textbf{access token} provided by the CoreIOT platform (e.g., \texttt{srQf3iXbZDdZjL8PuatC}). For a local broker, this can be any username configured in the broker's authentication system.
      \item \textbf{Password:} For CoreIOT cloud, the password field can be left empty or set to a placeholder (CoreIOT uses token-based auth). For a local MQTT broker with user/password authentication, this is the password corresponding to the configured username.
    \end{itemize}
  \item \textbf{Persisted Across Restarts:} Once configured, these settings are stored in the \texttt{Preferences} storage under the \texttt{WIFI\_STORE\_NAMESPACE}. They survive power cycles and Wi-Fi reconnections.
  \item \textbf{Reactive Reconnection:} If the user modifies the broker settings via the web interface, the CoreIOT task automatically detects the change. Upon the next configuration check (or upon reconnection), the task will disconnect from the old broker and connect to the newly specified broker.
\end{itemize}

This design allows seamless deployment flexibility:
\begin{itemize}
  \item In a \textbf{cloud-connected scenario}, the device publishes telemetry to the official CoreIOT platform (\texttt{app.coreiot.io}) using the CoreIOT access token.
  \item In a \textbf{private/local deployment scenario}, the device can be reconfigured via the web form to point to a local MQTT broker running on a private server, without any code changes or recompilation.
\end{itemize}

\subsection*{Telemetry Publishing and Payload Structure}

The CoreIOT task publishes rich, structured telemetry data every 10 seconds. The payload includes:
\begin{itemize}
  \item \textbf{Temperature (°C):} Real-time sensor reading from the DHT20.
  \item \textbf{Humidity (\%):} Real-time sensor reading from the DHT20.
  \item \textbf{Timestamp:} Unix epoch time (if NTP sync is available) or a tick-based fallback for relative timing. This enables accurate time-series data in the cloud dashboard.
  \item \textbf{Rain Probability (0-1):} Continuous probability output from the quantized TinyML model, ranging from 0.0 (no rain) to 1.0 (certain rain).
  \item \textbf{Rain Probability Percentage (0-100):} User-friendly percentage representation of the probability.
  \item \textbf{Categorical Rain Prediction:} Boolean classification ("RAIN" or "SUNNY") determined by the 0.5 threshold applied to the probability.
  \item \textbf{Short Rain Status String:} Human-readable forecast label such as \texttt{"Rain Expected"} or \texttt{"Safe Weather"}.
  \item \textbf{Current LCD Content Mode:} Indicates whether the local display is showing real-time sensor data or TinyML forecast. This can be controlled remotely.
  \item \textbf{Wi-Fi Connection State:} Signal strength (RSSI) and connection status, allowing the cloud dashboard to monitor device connectivity health.
\end{itemize}

This payload structure is carefully chosen because it balances data richness with network efficiency. Rather than sending only raw temperature and humidity, the system also transmits:
\begin{itemize}
  \item Derived analytics (TinyML prediction and probability),
  \item Contextual metadata (timestamp, display mode, signal strength), and
  \item Redundant representations (both 0-1 probability and 0-100 percentage).
\end{itemize}

This redundancy ensures that cloud dashboards and local logging systems have full context without requiring post-processing.

\subsection*{Remote Control and RPC Interaction}

The CoreIOT task is not purely a data publisher; it also acts as an \textbf{RPC responder}. The task subscribes to RPC request topics and can accept commands such as:
\begin{itemize}
  \item \textbf{Display Mode Toggle:} A cloud-side widget can send a command to switch the local LCD between real-time sensor display and TinyML forecast display.
  \item \textbf{Parameter Query:} Commands to request the current state, configuration, or other device metadata.
  \item \textbf{Control Signals:} Future extensions could include commands to adjust sampling intervals, change broadcast thresholds, or trigger firmware updates.
\end{itemize}

The RPC parser is designed to be flexible, accepting various boolean-like parameter formats (e.g., \texttt{"true"}, \texttt{"1"}, \texttt{"on"}) and mapping them to the appropriate local action. This approach enables a responsive, bidirectional cloud-to-device communication pattern without imposing a rigid command schema.

\subsection*{Failure Handling and Reconnection Logic}

The CoreIOT task implements robust failure recovery:
\begin{itemize}
  \item \textbf{Connection Loss Detection:} If the MQTT broker becomes unreachable, the task detects the disconnection and enters a reconnection loop.
  \item \textbf{Exponential Backoff:} Rather than attempting immediate reconnection (which could overwhelm the network), the task implements an exponential backoff strategy, with configurable retry intervals.
  \item \textbf{Graceful Degradation:} If cloud connection is unavailable, the local system continues to operate normally. Telemetry is not published, but sensor readings, LCD display, and local features remain fully functional.
  \item \textbf{Automatic Recovery:} Once the broker becomes reachable again, the task re-establishes connection and resumes publishing without manual intervention.
\end{itemize}

This design ensures that network transients do not compromise the integrity of the entire embedded system.


\subsection{MQTT Broker Implementation: TinyMQTT Service}

The MQTT broker is a standalone Python service designed to serve as an intermediary between the ESP32 IoT devices and the CoreIOT cloud platform. Unlike the firmware-side CoreIOT task (which acts as a client), the broker is a stateful service that accumulates telemetry from multiple devices, aggregates their data, and periodically forwards the aggregated stream to CoreIOT. The implementation demonstrates a complete IoT gateway pattern suitable for local deployment and demonstrates flexible authentication.

\subsection*{Architecture and Role}

The TinyMQTT broker serves three primary roles:
\begin{enumerate}
  \item \textbf{Local MQTT Broker:} It accepts incoming connections from multiple ESP32 devices using the standard MQTT protocol. Each device is authenticated using configured username and password credentials.
  \item \textbf{Data Aggregator:} It receives telemetry messages from each device, parses the JSON payloads, and stores them in an in-memory database indexed by device ID. This allows the broker to maintain a running history of samples.
  \item \textbf{Cloud Gateway:} It acts as an MQTT client to the CoreIOT platform, batching up accumulated telemetry and forwarding it at regular intervals (default: every 5 seconds) to \texttt{v1/gateway/telemetry} topic using a single authenticated connection.
\end{enumerate}

This three-layer pattern decouples the device communication (local MQTT) from the cloud platform communication (separate MQTT client), reducing latency on the device side and enabling flexible cloud platform switching without device code changes.

\subsection*{Configuration and Credentials}

The broker is configured through static settings at the top of \texttt{TinyMQTT.py}:
\begin{itemize}
  \item \textbf{BROKER\_IP:} The IP address where the broker listens for incoming device connections. Default: \texttt{192.168.137.1} (adjustable based on the host machine's network interface).
  \item \textbf{BROKER\_PORT:} The MQTT port, typically \texttt{1883} for standard MQTT.
  \item \textbf{BROKER\_USER and BROKER\_PASS:} Static credentials that all devices must use when connecting to the broker. These are hashed using SHA256 and stored in \texttt{password\_file.txt}.
  \item \textbf{GATEWAY\_HOST, GATEWAY\_PORT, GATEWAY\_ACCESS\_TOKEN:} Configuration for the CoreIOT cloud platform. The broker connects to \texttt{app.coreiot.io:1883} and authenticates with the provided access token. This token must be obtained from the CoreIOT admin dashboard.
  \item \textbf{GATEWAY\_PUBLISH\_INTERVAL\_SEC:} The interval (default 5 seconds) at which the broker batches and sends telemetry to CoreIOT.
\end{itemize}

All configuration is done at startup; no dynamic reconfiguration is currently supported, allowing the broker to be a simple, stateless service that can be containerized or deployed to a dedicated machine.

\subsection*{Device Auto-Detection and Dynamic Device Management}

Unlike pre-configured systems that require manual device registration, the TinyMQTT broker implements \textbf{automatic device discovery}:
\begin{itemize}
  \item \textbf{Topic-Based Device Identification:} Each incoming MQTT message is analyzed by its topic path. The broker expects messages published to topics of the form \texttt{devices/<device\_id>/telemetry}. The second segment of the topic is extracted as the \textbf{device ID}.
  \item \textbf{JSON Payload Parsing:} Each message is expected to be a JSON object containing sensor data. If the JSON includes a \texttt{device\_id} field, that value is used; otherwise, the device ID from the topic is retained.
  \item \textbf{Dynamic Device Creation:} The first time a message arrives from a new device ID, the broker automatically creates an entry for that device in the in-memory database. No manual registration, configuration file, or administrative action is required. Subsequent messages from the same device ID are appended to its growing sample list.
\end{itemize}

This approach offers several advantages:
\begin{itemize}
  \item \textbf{Zero Setup:} New devices can join the system by simply connecting to the broker and publishing to the standard topic pattern. They appear in the system immediately.
  \item \textbf{Scalability:} The broker can handle dozens or hundreds of devices without pre-configuration.
  \item \textbf{Flexibility:} Devices can dynamically change their device ID (though this is not recommended in practice) by publishing to a different topic path.
\end{itemize}

\subsection*{In-Memory Data Storage and Buffering}

The broker uses two key data structures to manage incoming telemetry:
\begin{itemize}
  \item \textbf{DEVICE\_SAMPLES:} A dictionary mapping device ID to a list of samples. Each sample is a dictionary containing \texttt{temperature}, \texttt{humidity}, \texttt{rain\_prob}, and \texttt{ts} (timestamp) fields. This structure persists indefinitely, maintaining a complete historical record of all samples received during the broker's uptime. It serves as a local data archive.
  \item \textbf{DEVICE\_PENDING:} A secondary dictionary also mapping device ID to samples. However, this list is cleared every time the gateway publisher sends telemetry to CoreIOT. Thus, DEVICE\_PENDING represents the "pending batch" of samples waiting to be forwarded to the cloud. This two-buffer pattern ensures that:
    \begin{itemize}
      \item Local archival is not lost when syncing to the cloud.
      \item The cloud publisher sends only new samples, avoiding redundant uploads.
    \end{itemize}
\end{itemize}

Both dictionaries are protected by \texttt{DEVICE\_LOCK}, a threading.Lock() object, ensuring thread-safe access when the subscriber thread is receiving messages and the gateway publisher thread is reading and clearing the pending buffer simultaneously.

\subsection*{Message Reception and Storage Workflow}

The on-device message reception is handled by the \texttt{run\_subscriber()} function, which runs in a dedicated thread:
\begin{enumerate}
  \item \textbf{Connection Establishment:} The subscriber creates an MQTT client, authenticates with \texttt{BROKER\_USER} and \texttt{BROKER\_PASS}, and connects to the local broker (127.0.0.1) at the configured port.
  \item \textbf{Topic Subscription:} The subscriber subscribes to the wildcard topic \texttt{\#}, which matches all incoming messages published to the broker.
  \item \textbf{Message Reception:} Upon receiving each message, the \texttt{on\_message()} callback is triggered. The payload is decoded as UTF-8 and parsed as JSON.
  \item \textbf{Device ID Extraction:} The topic path is split. If it matches the pattern \texttt{devices/<device\_id>/telemetry}, the device ID is extracted from the second segment. Otherwise, it attempts to read the device ID from the JSON \texttt{device\_id} field.
  \item \textbf{Sample Construction:} A dictionary containing the relevant fields (\texttt{temperature}, \texttt{humidity}, \texttt{rain\_prob}, \texttt{ts}) is built from the JSON payload.
  \item \textbf{Thread-Safe Storage:} Under mutex protection, the sample is appended to both \texttt{DEVICE\_SAMPLES[device\_id]} and \texttt{DEVICE\_PENDING[device\_id]}. If either list does not yet exist, it is created.
  \item \textbf{Logging:} A console message is printed indicating the device ID, total sample count, and the stored sample content. This provides real-time visibility into the broker's data ingestion.
\end{enumerate}

This workflow ensures that:
\begin{itemize}
  \item Every sample from every device is captured, parsed, and stored.
  \item Invalid JSON is gracefully rejected (with a warning message).
  \item Non-JSON payloads are also logged and skipped.
  \item Device IDs are dynamically detected and managed without pre-registration.
\end{itemize}

\subsection*{Batched Cloud Publishing to CoreIOT}

In parallel to the subscriber thread, the \texttt{start\_gateway\_publisher()} function runs in a separate thread and implements the cloud synchronization logic:
\begin{enumerate}
  \item \textbf{Cloud Connection:} The publisher creates a second MQTT client, authenticates using the \texttt{GATEWAY\_ACCESS\_TOKEN} as the username (CoreIOT uses token-based auth), and connects to \texttt{app.coreiot.io} at port 1883.
  \item \textbf{Periodic Batching:} Every \texttt{GATEWAY\_PUBLISH\_INTERVAL\_SEC} seconds (default 5 seconds), the publisher wakes up and checks if there are any pending samples.
  \item \textbf{Snapshot and Clear:} Under mutex protection, the publisher captures a snapshot of the current \texttt{DEVICE\_PENDING} dictionary and immediately clears it. This ensures that:
    \begin{itemize}
      \item Even if new samples arrive during the upload, they are not lost.
      \item The next batch will only contain samples received after the previous batch.
    \end{itemize}
  \item \textbf{Payload Construction:} For each device in the snapshot, the publisher iterates through its pending samples. For each sample:
    \begin{itemize}
      \item If the sample includes a \texttt{ts} field, it is converted to milliseconds (CoreIOT expects millisecond precision).
      \item If \texttt{ts} is missing, the current system time is used.
      \item A dictionary containing \texttt{ts} and \texttt{values} (temperature, humidity, rain\_prob) is appended to a per-device list.
    \end{itemize}
  \item \textbf{JSON Serialization and Upload:} The entire payload structure is serialized to JSON and published to the \texttt{v1/gateway/telemetry} topic on CoreIOT. The CoreIOT platform then deserializes this telemetry and associates each sample with the corresponding device ID.
  \item \textbf{Logging:} Upon successful publish, a message is logged indicating successful gateway delivery.
\end{enumerate}

The resulting payload structure sent to CoreIOT has the following form:
\begin{verbatim}
{
  "device_1": [
    {"ts": 1700000000000, "values": {"temperature": 25.3, "humidity": 60, "rain_prob": 0.15}},
    {"ts": 1700000005000, "values": {"temperature": 25.5, "humidity": 61, "rain_prob": 0.18}}
  ],
  "device_2": [
    {"ts": 1700000001000, "values": {"temperature": 28.1, "humidity": 55, "rain_prob": 0.05}}
  ]
}
\end{verbatim}

This structure enables CoreIOT to:
\begin{itemize}
  \item Track multiple devices in a single batch.
  \item Associate each sample with precise timestamps.
  \item Build time-series charts and analytics per device.
  \item Apply historical analysis and anomaly detection at the cloud level.
\end{itemize}

\subsection*{Multi-Threading and Synchronization}

The TinyMQTT broker is a multi-threaded application with three main threads:
\begin{enumerate}
  \item \textbf{Broker Thread:} Runs the hbmqtt broker engine, which accepts device connections and routes messages internally.
  \item \textbf{Subscriber Thread:} Runs the MQTT subscriber client, which listens to all messages on the broker and populates the in-memory data structures.
  \item \textbf{Gateway Publisher Thread:} Periodically batches and uploads telemetry to the CoreIOT cloud.
\end{enumerate}

All threads are daemon threads, meaning if the main process exits, they are terminated immediately without a graceful shutdown sequence. Thread safety is ensured through \texttt{DEVICE\_LOCK} protecting all access to \texttt{DEVICE\_SAMPLES} and \texttt{DEVICE\_PENDING}.

A careful reading of the code shows that the subscriber and publisher never simultaneously read and write to the same structures:
\begin{itemize}
  \item The \textbf{subscriber} continuously appends to both \texttt{DEVICE\_SAMPLES} and \texttt{DEVICE\_PENDING}.
  \item The \textbf{publisher} reads \texttt{DEVICE\_PENDING}, then clears it. It never modifies \texttt{DEVICE\_SAMPLES}.
\end{itemize}

This asymmetry means contention is minimized and deadlock risk is low. The lock is typically held for only a few microseconds per access.

\subsection*{Failure Modes and Robustness}

The broker implementation exhibits several desirable failure behaviors:
\begin{itemize}
  \item \textbf{Device Connection Loss:} If an ESP32 device disconnects, the broker continues operating. Reconnecting devices are treated as new clients and begin publishing immediately. No special recovery logic is needed.
  \item \textbf{Cloud Connectivity Loss:} If CoreIOT becomes unreachable, the subscriber continues to accept and buffer device messages. Samples accumulate in \texttt{DEVICE\_PENDING} until cloud connectivity is restored. Once the cloud connection recovers, buffered samples are sent in the next batch cycle.
  \item \textbf{JSON Parse Errors:} If a device publishes malformed JSON, the subscriber logs the error and skips the message. The broker remains operational.
  \item \textbf{Broker Restart:} If the broker service is restarted, in-memory buffers are lost (DEVICE\_SAMPLES and DEVICE\_PENDING are cleared). However, once devices reconnect and resume publishing, the system returns to normal operation. For permanent archival, an external database could be integrated.
\end{itemize}

This robustness makes the broker suitable for long-running deployments in environments with occasional network transients.

\subsection*{Extensibility and Future Enhancements}

The current TinyMQTT implementation is intentionally kept simple and modular. It is well-suited for future enhancements:
\begin{itemize}
  \item \textbf{Database Persistence:} Adding a SQL database (SQLite, PostgreSQL) to persist DEVICE\_SAMPLES beyond broker uptime.
  \item \textbf{Multiple Cloud Backends:} Supporting secondary cloud platforms (e.g., ThingsBoard, Azure IoT Hub) by instantiating additional gateway publisher clients.
  \item \textbf{Dynamic Configuration:} Implementing a REST API or web interface to reconfigure broker settings without restarting.
  \item \textbf{Rate Limiting and Quotas:} Adding per-device or per-message rate limits to prevent misbehaving devices from overwhelming the system.
  \item \textbf{Data Transformation Pipelines:} Inserting intermediate processing (e.g., anomaly detection, data aggregation) between ingestion and cloud publish.
  \item \textbf{MQTT Advanced Features:} Implementing QoS 2 (exactly-once delivery), retained messages, or last-will-and-testament (LWT) for enhanced reliability.
\end{itemize}

  \item converting epoch time into local date-time fields,

Although the current runtime path does not actively show a complete NTP setup

\subsection{Implementation Qualities}
Several implementation features are worth highlighting as strengths of the
project:
\begin{itemize}
  \item The system uses \texttt{xQueueOverwrite()} effectively for latest-value
  semantics.
  \item The sensing period is stabilized through \texttt{vTaskDelayUntil()}.
  \item Shared state is reduced and protected by a mutex.
  \item The LCD behavior is policy-driven rather than hard-coded to one screen.
  \item The web interface is not a static page; it actively supports monitoring,
  configuration, and control.
  \item The TinyML task is integrated into the same RTOS pipeline as the other
  system functions.
  \item The CoreIOT task supports both telemetry and remote RPC interaction.
\end{itemize}

Taken together, these implementation choices show that the system is not merely
feature-complete for the assignment; it is architecturally coherent.
